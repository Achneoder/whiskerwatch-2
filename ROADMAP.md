Kept as a record of how we got here. Full detail lives in git history; this is the summary.

- **Phase 1 â€” real session end to end.** Roster CRUD for player mice and hirelings; per-entity persisted stores (`persistedList.svelte.ts` + `storage.ts`); export/import of the whole campaign to a single JSON file (`campaignExport.ts`) so a campaign survives a browser wipe and can move between devices.

- **Phase 6 â€” Live Session tells the truth.** The header pulls the in-progress `Session` and active `Beat` from real stores instead of literal strings; the 1â€“2 faction clocks nearest to full are tappable pills right in the table view; hirelings appear alongside player mice with the same card component; the roll mechanic is a real Mausritter save (d20 roll-under a STR/DEX/WIL score, plain pass/fail, no crit/fumble rule).

Exceeding 10 slots shows an "Overburdened" warning banner that never blocks adding more.
- **Phase 8, wear/charge pips on items.** `tickCharge` (`lib/items.ts`) plus `tickMemberItemCharge`/`tickHirelingItemCharge` store wrappers let a charge track actually decrement, not just display. Deliberately no add/edit/remove/reslot in this surface â€” that stays prep-only in the Roster forms; Live Session is charge-ticking only.
- **Phase 8, hireling wages + loyalty saves.** `Hireling` gains a plain GM-entered `wage` (pips/day, no formula/auto-population from role). The IndexedDB migration flagged in the original Phase 1 was *not* done â€” the app is still `localStorage`-only. That's fine at current data volume; it's re-flagged as cross-cutting debt below because Phase 9 grows the volume and adds reference integrity concerns.

## Phase 9 â€” Weave the world together (âś… Done)

The integration debt, and the single biggest reason the app is four documents rather than one campaign tool. Beats, Hexes, Factions, and Bestiary are currently unrelated lists that share a color palette. Ranks as high as Phase 6 in GM value; it touches every screen a little rather than one screen a lot.

- **Beats reference hexes and factions. âś… Done.** `Beat` gained optional `hexNodeId: string | null` and `factionIds: string[]`. `BeatForm` edits both links (hex via a human-readable-label select, factions via an add/remove chip picker like `FactionForm`'s relationship picker); the relationship surfaces read-only on Hex Map ("beats touching this hex," in the hex detail modal) and Factions (as tags on the faction card and inside `FactionForm`), plus small tags under linked rows in `BeatTree`. Deleting a hex or faction cascades to null-out/strip the reference from any beat (`clearHexNodeFromBeats`, `removeFactionFromBeats` in `beats.svelte.ts`, called from `removeHexNode`/`removeFaction`) so a dangling ID can't crash the app, per the referential-integrity rule below. Legacy localStorage beats without these fields are backfilled on load. Covered by unit tests across all touched stores/components and a new `features/beat-links.feature` Playwright/Cucumber scenario.
- **Hex encounters draw from the real Bestiary. âś… Done.** `HexNode` gained `encounters: { bestiaryId, weight }[]`; the flat "d8 flavor text" `ENCOUNTER_TABLE` in `tables.ts` is gone, replaced by `lib/generators/encounters.ts`'s `generateEncounterFor`, which reads real `BestiaryEntry` records â€” weighted-random within a hex's linked creatures, or a uniform roll across the whole bestiary for "any hex"/hexes with no links. `HexNodeForm` gets an Encounters section (add/remove + in-place weight edit via a new `Stepper` `sm` size); the hex detail modal surfaces the list read-only as tags. A new `StatBlock` component (extracted from `Bestiary.svelte`, now shared) renders the rolled creature's real stat block in Generators instead of a sentence. Deleting a bestiary entry cascades to strip it from every hex's encounter list (`removeBestiaryEntryFromHexNodes`), and a dangling id already in storage degrades to an em-dash rather than crashing, per the referential-integrity rule below. Legacy hex records without `encounters` are backfilled on load. Covered by unit tests across all touched stores/components plus `features/hex-bestiary.feature`.
- **Hexes link to factions. âś… Done.** `HexNode` gained `controlledBy: string | null` and `contestedBy: string[]`. `HexNodeForm` sets a controller (single-select, auto-demotes from `contestedBy` if promoted) and toggles contesters (chip add/remove, mirroring the Encounters/Beats pickers); `HexCanvas` draws territory rings via `territoryRings(node)` â€” a solid ring for the controller and dashed rings for contesters, ordered hostileâ†’neutralâ†’ally so hostile claims read as the outermost/most urgent ring â€” reusing `FactionGraph`'s `dispositionRingColor` palette. The hex detail modal shows read-only "Controlled by"/contester tags via `dispositionTagTone`. Deleting a faction cascades via `removeFactionFromHexNodes` (nulls `controlledBy`, strips from `contestedBy`), and legacy hex records are backfilled with `controlledBy: null, contestedBy: []` on load. Covered by unit tests in `hexmap.test.ts`/`factions.test.ts`/`HexNodeForm.test.ts`/`HexMap.test.ts` and `features/hex-factions.feature`.
- **Generated content is keepable. âś… Done.** Only the NPC generator needed this â€” the Encounter generator already returns an existing `BestiaryEntry` (nothing new to save), and the Item generator's free text has no natural home in either store. The NPC result card gets "Save to Roster" / "Save to Bestiary" buttons; each opens the existing `HirelingForm`/`BestiaryForm` modal pre-filled with sensible defaults (stats at the standard "new hireling" baseline, `quirk`/`want` folded into `notes`) so the GM can adjust before committing rather than writing placeholder stats straight into a permanent record. On save the button flips to a disabled "Saved" state (the other stays active â€” the same NPC can go to both stores) and an inline undo notice appears, reusing `LiveSession.svelte`'s `Notice`/`announce()` pattern rather than a new toast component. Rolling again resets both save-states. `addHireling`/`addBestiaryEntry` now return the new id (previously `void`) so undo can remove the just-created record. Covered by new `Generators.test.ts` cases and `features/generator-save.feature`.

## Phase 10 â€” Close the loop: sessions that write themselves

Prep and live-play are covered by 6â€“9; this phase is the *after* â€” `CLAUDE.md`'s "next session starts fast."

- **Session auto-recap draft. âś… Done.** A new session-scoped, non-persisted event log (`liveSessionLog.svelte.ts`) records notable facts as they happen during Live Session â€” beat status changes, faction clock moves, deaths, Fatal Wounds scars gained, conditions gained, STR-overflow drains, advancements, and Loyalty save failures (only failures; passes and reversible/noisy events like individual damage or charge burns are deliberately not logged). Tapping a new "End Session" button (`LiveSessionHeader`) opens `SessionRecapReview`, a full-screen checklist grouping events under Beats/Factions/Party/Hirelings with pre-checked rows the GM can toggle off; "Draft Recap â†’" compiles only the checked facts into rules-accurate bullet lines ("Resolved: '...'", "reached level N", "gained the scar '...'", "STR drained to N", "failed a Loyalty save" â€” never "leveled up"/"took damage"/"wound") and routes into the existing `SessionForm` pre-filled (`summary` = bullets + a trailing blank line for the GM's own prose). `SessionForm` shows a dismissible "Auto-generated draft" pill that disappears the moment the GM edits the text, via a new optional `draft` prop kept separate from the existing `initial` (edit) prop. `Session.summary` stays a plain string â€” no new structured fields. Beat-status-change and scar-gain events are logged wherever those mutations already happen, but neither is reachable from the live-play UI yet (beat status only changes from the Adventure screen; there's no scar-adding UI at all), so those two event kinds exist in the log's type but can't yet be exercised end-to-end â€” a gap to close if/when those surfaces get live-session entry points. Covered by unit tests (`liveSessionLog.test.ts`, `SessionRecapReview.test.ts`, updated `SessionForm`/`Sessions`/`LiveSession` tests) and `features/session-recap.feature`.
- **Campaign timeline view. âś… Done.** A new persisted append-only `campaignHistory.svelte.ts` store logs `session`/`beatCompleted`/`clockChanged` entries at the source of truth (`sessions.svelte.ts`'s `addSession`, `beats.svelte.ts`'s `updateBeat` on a not-doneâ†’done transition, `factions.svelte.ts`'s `bumpFactionClock` when the value actually changes) â€” it's forward-only, not backfilled for pre-existing data. A new `Timeline` screen (nav entry after Sessions) renders it as a newest-first, day-grouped read-only feed with three toggleable type filters and tinted icon rows per entry type, reusing `Card`/`Tag`/`StatusPill`. Covered by unit tests across all touched stores plus `Timeline.test.ts`, and `features/timeline.feature`.
- **Scars & death ledger.** A small persisted list of permanent Fatal Wounds outcomes per mouse, shown on the roster card. Cheap since Phase 7's Fatal Wounds prompt already exists, and good for table drama.

## Cross-cutting, not a phase

- **IndexedDB migration.** Re-flagged from the original Phase 1; still not done, still fine for now (`persistedList.svelte.ts` on `localStorage` handles current volume). Revisit once Phase 8's item store and Phase 9's cross-links land â€” not speculatively before.
- **Referential integrity on delete.** `factions`/`factionEdges` already do this right (`removeEdgesForFaction`). Every Phase 9 cross-link (beatâ†’hex, hexâ†’faction, hexâ†’bestiary) needs the same cascade-or-null-out discipline, or deleting a hex silently orphans a beat's reference â€” and per `CLAUDE.md` a dangling ID must degrade gracefully, not crash.
- **Testing.** Every item keeps the established Vitest + Playwright/Cucumber pairing. The save mechanic especially deserves a Gherkin scenario â€” e.g. "Given Wren has 0 HP and a STR save fails, When the GM applies 3 more damage, Then Wren's STR drops and a Fatal Wounds prompt appears" â€” since it's a rules-correctness surface, not just UI.

## Explicitly out of scope (would be scope creep)

- **Full character-creation wizard** (rolling starting stats/background/items). GMs in this campaign already have their mice â€” this is new-campaign onboarding, not running-this-campaign. Revisit only if zero-to-one setup becomes a target use case.
- **Weather/season/calendar systems.** Mausritter has light seasonal downtime rules, but nothing at this table has asked for it. Don't build ahead of real need.
- **Multi-campaign / co-GM real-time sync.** No backend by design (`CLAUDE.md`); the "hand off to a co-GM" story is the export/import flow already shipped, not sync.
