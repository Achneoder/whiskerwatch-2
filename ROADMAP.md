# Whiskerwatch Roadmap

Where the app is, and what's next, ordered by what actually unblocks running a real campaign.

## Where we are

**Phases 1–11 have shipped.** Every screen in `CLAUDE.md`'s domain list exists and is wired to persisted, IndexedDB-backed stores: Dashboard (now with a Session Prep checklist), Roster (parties/mice/hirelings), Adventure (beat tree), Bestiary, Factions (relationship graph + clocks), Hex Map (canvas + per-hex nodes), Generators (dice/encounter/item/NPC/reaction), Sessions (log/recap), Settings, Timeline (chronological campaign feed), and the Live Session table view. Live Session reads real state (session/beat/faction/hireling data, no fixture strings) and rolls actual Mausritter mechanics: a d20 roll-under save, a 2d6 loyalty save, and a flat 2d6 reaction roll. Player mice and hirelings carry real STR/DEX/WIL, the HP→STR damage-overflow death spiral, the standardized condition vocabulary, downtime XP, a rules-accurate 10-slot item inventory, a wage + loyalty save, GM-narrated permanent scars, and a persisted death ledger. A quick-reference rules drawer covers saves/damage/conditions/inventory/reactions/advancement/retainer-limits without leaving the app. The design system (tokens, `ui/` primitives), export/import of the whole campaign to JSON, a PWA manifest + service worker, and English/German i18n via `svelte-i18n` are all in place. Phase 9 wove the previously-siloed screens together (beats↔hexes/factions, hex encounters from the real Bestiary, hex↔faction territory, keepable generated NPCs); Phase 10 closed the loop on session wrap-up (auto-recap drafts, the Timeline feed, scars/death); Phase 11 closed out missing rules coverage and storage debt (reaction rolls/retainer limits, the rules drawer, the IndexedDB migration, a referential-integrity audit, and the Dashboard prep checklist).

**Two features shipped after Phase 11 was marked complete, without ever being scoped as a phase.** Both are real, both are done, and both are described here for the first time — see "A note on unscoped work" below for why they're documented this way instead of as a retroactive "Phase 11.5."

- **Multiple concurrent adventures, each with its own beat tree.** What used to be a single implicit root `Beat` (the one hard-coded "granary raid" outline) is now a real `Adventure` entity (`lib/stores/adventures.svelte.ts`: `title`/`description`/`status` of `planned`/`active`/`completed`) that owns its own beat tree — every `Beat` gained an `adventureId` (`lib/stores/beats.svelte.ts`), and the Adventure screen renders one card per adventure, each with its own independent `BeatTree`. A one-time, idempotent migration (`migrateLegacyBeatsToAdventures`) promotes any pre-existing root beat into its own `Adventure` on first boot, so long-running campaigns upgrade losslessly. This is a real GM need — parallel plot threads (the granary raid *and* the simmering feud with the Gnawing Court) are normal in an ongoing hex-crawl campaign, not an edge case.
- **Campaign renaming.** `lib/stores/campaign.svelte.ts` holds a trivial `{ name: string }` `CampaignSettings`, editable inline from the Dashboard header (click the title, edit, blur/enter to save), backed by `localStorage` (correctly, per `CLAUDE.md` — this is a small setting, not structured campaign data needing IndexedDB's integrity guarantees). It round-trips through export/import (`campaignExport.ts` reads and restores `campaignName`). Small, complete, no follow-up needed.

Phases 1–11 are summarised under "Shipped" below and kept for history.

## Shipped (Phases 1–9)

Kept as a record of how we got here. Full detail lives in git history; this is the summary.

- **Phase 1 — real session end to end.** Roster CRUD for player mice and hirelings; per-entity persisted stores (`persistedList.svelte.ts` + `storage.ts`); export/import of the whole campaign to a single JSON file (`campaignExport.ts`) so a campaign survives a browser wipe and can move between devices.
- **Phase 2 — prep tools.** Adventure beat tree (`BeatTree`, `BeatForm`) and a Sessions log (`SessionForm`, `sessions.svelte.ts`).
- **Phase 3 — bestiary & generators.** Rules-accurate Bestiary stat blocks; dice roller and encounter/item/NPC generators (`lib/generators/tables.ts`).
- **Phase 4 — factions & hex map.** Faction list + relationship graph (`FactionGraph`) + clocks; hex-crawl canvas (`HexCanvas`) with per-hex nodes (`HexNodeForm`).
- **Phase 5 — polish & cross-cutting.** Playwright + Cucumber BDD harness stood up; per-screen i18n (English/German); Settings screen wired to theme/language/export-import; PWA manifest + service worker (`lib/pwa.ts`) for offline use at the table.
- **Phase 6 — Live Session tells the truth.** The header pulls the in-progress `Session` and active `Beat` from real stores instead of literal strings; the 1–2 faction clocks nearest to full are tappable pills right in the table view; hirelings appear alongside player mice with the same card component; the roll mechanic is a real Mausritter save (d20 roll-under a STR/DEX/WIL score, plain pass/fail, no crit/fumble rule).
- **Phase 7 — mice have the stats Mausritter runs on.** STR/DEX/WIL on `PartyMember`/`Hireling`; HP-then-STR damage overflow (`combat.ts`'s `applyDamage`) with the real death spiral (STR save on any STR drain, Injured+Incapacitated on a fail, immediate death at exactly 0 STR — no random Fatal Wounds table, just a one-tap death confirm); the standardized condition vocabulary (`lib/conditions.ts`) with a tap-to-add chip picker; downtime XP from pips spent, not kills, with the real SRD advancement thresholds.
- **Phase 8, item model + slot inventory.** A rules-accurate fixed 10-slot inventory (6 body + 4 paws, one flat array — the split is a UI convention, not two storage pools) on both `PartyMember` and `Hireling` (`lib/items.ts` + thin per-store wrappers, mirroring `combat.ts`'s shared-pure-function pattern). Tap-to-edit slot grid (`ItemSlotGrid.svelte`) in the Roster edit forms; 2-slot bulky items span a cell; a 6-pip charge track renders as display-only dots. Exceeding 10 slots shows an "Overburdened" warning banner that never blocks adding more.
- **Phase 8, wear/charge pips on items.** `tickCharge` (`lib/items.ts`) plus `tickMemberItemCharge`/`tickHirelingItemCharge` store wrappers let a charge track actually decrement, not just display. Live Session gets its first inventory surface: a "Bag N/10" pill on each `LiveSessionCard` opens `LiveSessionInventoryModal`, a fixed 2-column paws/body grid where tapping a chargeable item burns its next pip (whole-cell tap target, not the tiny pip glyphs) with the same undo-toast pattern as damage/conditions. Deliberately no add/edit/remove/reslot in this surface — that stays prep-only in the Roster forms; Live Session is charge-ticking only.
- **Phase 8, hireling wages + loyalty saves.** `Hireling` gains a plain GM-entered `wage` (pips/day, no formula/auto-population from role). Live Session gets a tappable Loyalty pill (on the hireling's card, and as a 4th "LOY" button in `SaveDock`) that rolls the real Mausritter loyalty save — 2d6 roll-under Loyalty (2–12 range), a *different* primitive from the d20 `rollSave` used for STR/DEX/WIL (`rollLoyaltySave` in `lib/generators/save.ts`), reported pass/fail only with no auto-decrement — the GM narrates the consequence of a failed save. A "Pay Day" modal lists active hirelings with their wage and a tap-to-toggle Paid/Unpaid tag (deliberately local, unpersisted state that resets every reopen — not a ledger) plus an inline loyalty-check shortcut for anyone still unpaid when the GM closes out. **Deviation from the original roadmap wording:** non-payment does *not* auto-decrement a shared pip purse — a `gm-product-owner`/rules pass on this bullet confirmed Mausritter has no such mechanical penalty, so Pay Day is bookkeeping + a rolling shortcut only, never an automatic mutation.

The IndexedDB migration flagged in the original Phase 1 was *not* done — the app is still `localStorage`-only. That's fine at current data volume; it's re-flagged as cross-cutting debt below because Phase 9 grows the volume and adds reference integrity concerns.

## Phase 9 — Weave the world together (✅ Done)

The integration debt, and the single biggest reason the app is four documents rather than one campaign tool. Beats, Hexes, Factions, and Bestiary are currently unrelated lists that share a color palette. Ranks as high as Phase 6 in GM value; it touches every screen a little rather than one screen a lot.

- **Beats reference hexes and factions. ✅ Done.** `Beat` gained optional `hexNodeId: string | null` and `factionIds: string[]`. `BeatForm` edits both links (hex via a human-readable-label select, factions via an add/remove chip picker like `FactionForm`'s relationship picker); the relationship surfaces read-only on Hex Map ("beats touching this hex," in the hex detail modal) and Factions (as tags on the faction card and inside `FactionForm`), plus small tags under linked rows in `BeatTree`. Deleting a hex or faction cascades to null-out/strip the reference from any beat (`clearHexNodeFromBeats`, `removeFactionFromBeats` in `beats.svelte.ts`, called from `removeHexNode`/`removeFaction`) so a dangling ID can't crash the app, per the referential-integrity rule below. Legacy localStorage beats without these fields are backfilled on load. Covered by unit tests across all touched stores/components and a new `features/beat-links.feature` Playwright/Cucumber scenario.
- **Hex encounters draw from the real Bestiary. ✅ Done.** `HexNode` gained `encounters: { bestiaryId, weight }[]`; the flat "d8 flavor text" `ENCOUNTER_TABLE` in `tables.ts` is gone, replaced by `lib/generators/encounters.ts`'s `generateEncounterFor`, which reads real `BestiaryEntry` records — weighted-random within a hex's linked creatures, or a uniform roll across the whole bestiary for "any hex"/hexes with no links. `HexNodeForm` gets an Encounters section (add/remove + in-place weight edit via a new `Stepper` `sm` size); the hex detail modal surfaces the list read-only as tags. A new `StatBlock` component (extracted from `Bestiary.svelte`, now shared) renders the rolled creature's real stat block in Generators instead of a sentence. Deleting a bestiary entry cascades to strip it from every hex's encounter list (`removeBestiaryEntryFromHexNodes`), and a dangling id already in storage degrades to an em-dash rather than crashing, per the referential-integrity rule below. Legacy hex records without `encounters` are backfilled on load. Covered by unit tests across all touched stores/components plus `features/hex-bestiary.feature`.
- **Hexes link to factions. ✅ Done.** `HexNode` gained `controlledBy: string | null` and `contestedBy: string[]`. `HexNodeForm` sets a controller (single-select, auto-demotes from `contestedBy` if promoted) and toggles contesters (chip add/remove, mirroring the Encounters/Beats pickers); `HexCanvas` draws territory rings via `territoryRings(node)` — a solid ring for the controller and dashed rings for contesters, ordered hostile→neutral→ally so hostile claims read as the outermost/most urgent ring — reusing `FactionGraph`'s `dispositionRingColor` palette. The hex detail modal shows read-only "Controlled by"/contester tags via `dispositionTagTone`. Deleting a faction cascades via `removeFactionFromHexNodes` (nulls `controlledBy`, strips from `contestedBy`), and legacy hex records are backfilled with `controlledBy: null, contestedBy: []` on load. Covered by unit tests in `hexmap.test.ts`/`factions.test.ts`/`HexNodeForm.test.ts`/`HexMap.test.ts` and `features/hex-factions.feature`.
- **Generated content is keepable. ✅ Done.** Only the NPC generator needed this — the Encounter generator already returns an existing `BestiaryEntry` (nothing new to save), and the Item generator's free text has no natural home in either store. The NPC result card gets "Save to Roster" / "Save to Bestiary" buttons; each opens the existing `HirelingForm`/`BestiaryForm` modal pre-filled with sensible defaults (stats at the standard "new hireling" baseline, `quirk`/`want` folded into `notes`) so the GM can adjust before committing rather than writing placeholder stats straight into a permanent record. On save the button flips to a disabled "Saved" state (the other stays active — the same NPC can go to both stores) and an inline undo notice appears, reusing `LiveSession.svelte`'s `Notice`/`announce()` pattern rather than a new toast component. Rolling again resets both save-states. `addHireling`/`addBestiaryEntry` now return the new id (previously `void`) so undo can remove the just-created record. Covered by new `Generators.test.ts` cases and `features/generator-save.feature`.

## Phase 10 — Close the loop: sessions that write themselves

Prep and live-play are covered by 6–9; this phase is the *after* — `CLAUDE.md`'s "next session starts fast."

- **Session auto-recap draft. ✅ Done.** A new session-scoped, non-persisted event log (`liveSessionLog.svelte.ts`) records notable facts as they happen during Live Session — beat status changes, faction clock moves, deaths, Fatal Wounds scars gained, conditions gained, STR-overflow drains, advancements, and Loyalty save failures (only failures; passes and reversible/noisy events like individual damage or charge burns are deliberately not logged). Tapping a new "End Session" button (`LiveSessionHeader`) opens `SessionRecapReview`, a full-screen checklist grouping events under Beats/Factions/Party/Hirelings with pre-checked rows the GM can toggle off; "Draft Recap →" compiles only the checked facts into rules-accurate bullet lines ("Resolved: '...'", "reached level N", "gained the scar '...'", "STR drained to N", "failed a Loyalty save" — never "leveled up"/"took damage"/"wound") and routes into the existing `SessionForm` pre-filled (`summary` = bullets + a trailing blank line for the GM's own prose). `SessionForm` shows a dismissible "Auto-generated draft" pill that disappears the moment the GM edits the text, via a new optional `draft` prop kept separate from the existing `initial` (edit) prop. `Session.summary` stays a plain string — no new structured fields. Beat-status-change and scar-gain events are logged wherever those mutations already happen, but neither is reachable from the live-play UI yet (beat status only changes from the Adventure screen; there's no scar-adding UI at all), so those two event kinds exist in the log's type but can't yet be exercised end-to-end — a gap to close if/when those surfaces get live-session entry points. Covered by unit tests (`liveSessionLog.test.ts`, `SessionRecapReview.test.ts`, updated `SessionForm`/`Sessions`/`LiveSession` tests) and `features/session-recap.feature`.
- **Campaign timeline view. ✅ Done.** A new persisted append-only `campaignHistory.svelte.ts` store logs `session`/`beatCompleted`/`clockChanged` entries at the source of truth (`sessions.svelte.ts`'s `addSession`, `beats.svelte.ts`'s `updateBeat` on a not-done→done transition, `factions.svelte.ts`'s `bumpFactionClock` when the value actually changes) — it's forward-only, not backfilled for pre-existing data. A new `Timeline` screen (nav entry after Sessions) renders it as a newest-first, day-grouped read-only feed with three toggleable type filters and tinted icon rows per entry type, reusing `Card`/`Tag`/`StatusPill`. Covered by unit tests across all touched stores plus `Timeline.test.ts`, and `features/timeline.feature`.
- **Scars & death ledger. ✅ Done.** Scars are GM-narrated free text (`ScarForm.svelte`: 6 tappable suggestion chips that fill, not submit, a label + optional note), not rolled from Mausritter's SRD Fatal Wounds table — a deliberate simplification consistent with Phase 7's dropped random table. Fully manual, never gated on a specific mechanical event; `+ Scar` lives directly on the Roster row (`ScarRow.svelte`, shared by party members and hirelings, showing up to 3 scar tags plus a `+N` overflow modal). `confirmDeath` in Live Session gained an optional "Cause of death" input (never blocks the confirm) and now calls `campaignHistory.svelte.ts`'s new `logDeath` alongside the existing `killMember`/`killHireling`, so deaths get a persisted ledger entry (name, role, source, cause, session, level) and a dedicated Skull-icon row on the Timeline screen. Deceased mice move into a collapsed "Fallen (N)" section on Roster, mirroring Live Session's existing fallen-hirelings pattern. Covered by unit tests across all touched stores/components (`ScarForm.test.ts`, `ScarRow.test.ts`, updated `Roster`/`LiveSession`/`Timeline`/`campaignHistory` tests) and `features/scars-and-death.feature`.

## Phase 11 — Rules coverage, data durability, and dead-air reduction

Phases 1–10 built and wove together every screen the domain needs. Phase 11 isn't new screens — it's closing gaps that only became visible once the app was mature: rules mechanics that fire every session but aren't modeled, storage debt that was correctly deferred twice but no longer is, and small conveniences that compound with what already exists. Ranked by GM value.

- **Reaction rolls + retainer/warband limits. ✅ Done.** `lib/generators/reaction.ts`'s `rollReaction()` is a flat 2d6→SRD band lookup (Hostile 2 / Unfriendly 3–5 / Neutral 6–8 / Friendly 9–11 / Helpful 12, each with its SRD guidance sentence) — a deliberately different shape from `rollLoyaltySave`'s roll-under-save, since a reaction roll has no pass/fail concept. Surfaced via a shared `ReactionResult.svelte` (built on `DiceRoll.svelte`, extended with hostile/unfriendly/friendly/helpful tones) in both the Generators Encounter tab (under the rolled `StatBlock`) and a new `LiveSessionEncounterCard.svelte` in Live Session, which only appears when the active beat has a linked hex (Phase 9's beat↔hex link) — Live Session previously had no hex-encounter surface at all, so this card's "Roll an encounter" step was net-new scaffolding, not a pre-existing hook. Rerolling an encounter clears any prior reaction roll on both surfaces. The retainer limit (a mouse's WIL = hirelings it can command, summed across the whole party — not a per-mouse assignment) surfaces as a non-blocking warning banner on Roster's Hirelings card when the count is exceeded, reappearing/disappearing reactively as the roster changes; it never gates hiring. Covered by unit tests across all touched components/stores plus `features/reaction-roll.feature`.
- **Quick-reference rules drawer. ✅ Done.** `RulesDrawer.svelte` is a true slide-out panel (not a `Modal` — scrim + edge-anchored panel, full-width sheet on phone, 420px on desktop), opened via a `BookOpen` trigger in `LiveSessionHeader` and `Generators`' header. Seven sections (saves, damage/death spiral, conditions, inventory/slots, reaction roll, advancement, retainer limits) stacked in one scrollable column with a sticky jump-strip for quick navigation — no accordion/tab state to fumble under table pressure. Content was verified against the real constants before writing (`lib/conditions.ts`'s condition vocabulary, `party.svelte.ts`'s `DOWNTIME_XP_PER_PIP`/`XP_THRESHOLDS`) rather than restated from memory, and reuses the reaction-band text from Phase 11's reaction-roll feature instead of duplicating it. Read-only, static content, no new data model. Covered by `RulesDrawer.test.ts` plus updated `LiveSession`/`LiveSessionHeader`/`Generators` tests and `features/rules-drawer.feature`.
- **IndexedDB migration (promote from cross-cutting debt). ✅ Done.** All 9 `createPersistedList`-backed stores (party, hirelings, beats, sessions, bestiary, factions, factionEdges, hexmap, campaignHistory) now live in IndexedDB instead of `localStorage`; `theme.svelte.ts`/i18n's locale deliberately stay on `localStorage` (tiny synchronous preferences, no volume/reference-integrity concerns). `lib/idb.ts` wraps the native API around one shared `items` object store (keyed by `${listKey}::${id}`, indexed by `listKey`) plus a `meta` store tracking per-list "initialized" state, so a deliberately-emptied list never silently reseeds. `persistedList.svelte.ts` keeps its fully synchronous public surface (`items`/`add`/`update`/`remove`/`replaceAll`, plus a new `flush()` for durability-sensitive callers) — `items` starts empty and is populated once hydration's `ready` promise resolves; `main.ts` awaits every store's `ready` (via `allStoresReady`) behind a brief `BootLoading` spinner before mounting `App.svelte`, so every existing call site still sees already-populated data. Legacy `localStorage` data migrates in-place on first boot and the old key is retired. Mutators are optimistic (in-memory update first) with fire-and-forget background writes; screens whose "save"/"delete" close a modal (Roster, Adventure, Bestiary, Factions, HexMap, Sessions) now `await flush()` first so a GM who refreshes immediately after saving never loses the edit — this was a real regression the e2e suite caught (`fake-indexeddb`'s Node `structuredClone` also can't clone Svelte 5's `$state` proxies directly, worked around with `$state.snapshot()` before every write). Test setup imports `fake-indexeddb/auto` and awaits `allStoresReady` in a global `beforeAll` so the ~450 existing unit tests keep their synchronous-looking assertions unchanged. Covered by `idb.test.ts`, an updated `persistedList.test.ts`, and the full existing Playwright/Cucumber suite passing unmodified in behavior.
- **Referential-integrity audit pass (promote from cross-cutting debt). ✅ Done.** A dedicated pass across all seven cross-link surfaces (beat↔hex/faction, hex↔bestiary, hex↔faction territory, `campaignHistory`→session/beat/faction/member ids, scars/death→party-member, the Phase 11 encounter card's beat→hex lookup, the reaction roll/WIL-limit banner) found every cascade-or-null-out path already correct — `removeHexNode`/`removeFaction`/`removeBestiaryEntry` already cascade as designed, `campaignHistory` deliberately snapshots display data inline rather than doing live lookups (so it never dereferences a stale id at all) except `beatCompleted`'s hex/faction tags, which already resolve dangling ids to nothing rather than crashing, and `LiveSession`'s beat→hex resolution already degrades to hiding the encounter card. The only real gap was missing regression tests proving two of those already-correct paths — closed with new cases in `Timeline.test.ts` and `LiveSession.test.ts`. No production code changes were needed.
- **Session prep checklist. ✅ Done.** A new "Session Prep" `Card` on Dashboard, between the stat grid and the Warband/Factions grid: four tappable rows — active beats, hexes with encounters configured, active wage-drawing hirelings (a reminder to check Pay Day, not a computed "unpaid" list, since Phase 8 deliberately never persisted a payment ledger), and faction clocks near-full (reusing Live Session's exact `nearFull` threshold rather than a new one). Zero counts render as "0" rather than hiding the row; only when all four are simultaneously zero does the panel collapse to a single reassuring "Nothing urgent — you're ready to run." line. Tapping a row navigates to the relevant screen. All read from existing store data, no new persistence. Covered by an extended `Dashboard.test.ts` and `features/dashboard.feature`.

**Explicitly not in Phase 11:** printable/exportable handouts (nobody's asked, and export-to-JSON already covers "hand this to someone else"); expanding downtime activities beyond XP (Mausritter's downtime rules are optional and this table hasn't hit that need). These stay off the list rather than getting queued — revisit only on real demand, same bar as the out-of-scope section below.

## Phase 12 — Make concurrent adventures actually work at the table (✅ Done)

The multi-adventure feature landed as a prep-side data model change (Adventure screen, beat tree, migration) but nothing downstream of it was updated to know adventures can now be plural. That's fine as long as a GM only ever has one adventure with an active beat at a time — but the entire point of shipping "concurrent" adventures is to support two live plot threads at once, and the moment a table actually does that, Live Session gets it wrong silently. Phase 12 closes that gap and tidies the one prep-side consequence (list growth) before it becomes annoying. This is a small, focused phase — not a new domain concept, just finishing the last one properly.

- **Live Session's active-beat lookup is adventure-blind — this is a correctness bug, not polish. (Must-have.) ✅ Done.** `LiveSession.svelte` derived the table's active beat with `getBeats().find((b) => b.status === 'active')` — a single global lookup with no `adventureId` filter, so two concurrent adventures each carrying their own active beat could silently surface the wrong hex/encounter context to the GM. Fixed per a `ux-ui-designer` spec (`docs/design/phase-12-live-session-adventure-picker.md`): `LiveSession.svelte` now derives `activeAdventureOptions` across all adventures with an active beat, and `LiveSessionHeader.svelte` renders a chip + inline listbox picker (`aria-haspopup="listbox"`, `role="listbox"`/`role="option"`, 44px tap targets) *only* when 2+ distinct adventures are active — in the 0/1 case the picker renders nothing and behavior is byte-for-byte the same as before. Selection is transient `$state`, not persisted, and now also feeds the End Session recap's `adventureId` (see the third bullet below). Covered by `LiveSessionHeader.test.ts`, `LiveSession.test.ts`, and `features/adventure-picker.feature`.
- **Adventure screen needs to collapse completed adventures. (Must-have, and cheap given #1 sets the pattern.) ✅ Done.** `Adventure.svelte` now splits adventures into `liveAdventures`/`completedAdventures` and renders completed ones inside a `<details>`/`<summary>` "Completed (N)" section at the bottom, mirroring Roster's Fallen-mice pattern exactly — same collapsed-by-default, expandable interaction the GM already knows. Covered by new cases in `Adventure.test.ts` and `features/adventure-completed.feature`.
- **Sessions should be taggable to an adventure. (Nice-to-have.) ✅ Done.** `Session` gained an optional `adventureId?: string | null` (`sessions.svelte.ts`). `SessionRecapReview`/`SessionForm` default it from whichever adventure the Live Session picker (first bullet) was scoped to when "End Session" was tapped, with a manual `<select>` in `SessionForm` (rendered only when adventures exist) if the GM wants to override. Sessions and Timeline screens both get a single-select filter chip (All / per-adventure / Unassigned); Timeline's filter only narrows `session`-type entries, since no other history-entry kind carries an `adventureId` yet — an intentional scope limit, not a bug. Legacy sessions without the field simply don't match any adventure filter; no backfill migration needed. Covered by `sessions.test.ts`, `SessionForm.test.ts`, `SessionRecapReview.test.ts`, `Sessions.test.ts`, `Timeline.test.ts`.

Verified: `pnpm typecheck` clean (0 errors), 516/516 unit tests passing, and 37/39 Playwright/Cucumber scenarios passing — the 2 failures (`dashboard.feature`'s Session Prep panel, `timeline.feature`'s beat-completion scenario) were reproduced identically against unmodified `main` and are pre-existing flakes unrelated to this phase.

**Explicitly not in Phase 12:** auto-syncing `Adventure.status` from its beats' statuses (e.g. auto-completing an adventure when its last beat is marked done). The one-time migration maps beat status to adventure status *once*, on promotion, and `AdventureForm` has let the GM set status independently ever since — that's correct, not a bug: an adventure can be narratively "wrapping up" while a mop-up beat is still technically open, and forcing the two to stay mechanically linked would take a GM judgment call away for no real benefit. Leave it manual. Also not in scope: per-adventure hireling assignment — Mausritter's retainer limit (WIL = hirelings commandable) is a party-wide constraint, not a per-plot-thread one, and Phase 11 already modeled it correctly at the party level.

## Phase 13 — Finish what Live Session started with encounters and inventory

Phase 11 gave Live Session an encounter card and a charge-ticking inventory modal, but both stop short of the moment they matter most: a rolled encounter has no way to track HP once the fight actually starts, and there's no way to hand an item from one mouse to another mid-session. Both are correctness gaps in features that already shipped, not new domain concepts — the same shape of problem Phase 12 solved for concurrent adventures. Ranked by GM value.

- **Live-combat HP tracker for rolled encounters. (Must-have.)** `LiveSessionEncounterCard.svelte` rolls a creature and shows its static `StatBlock` with no way to track damage — the GM has to fall back to paper the moment a fight starts, in an app whose whole premise is running the table from here. Needs: rolling an encounter spawns one or more tracked instances (supporting groups, e.g. "3 Ratlings," not just solo creatures), each damageable the same way `LiveSessionCard` already damages a mouse, with a mark-dead/remove action once an instance is defeated. Route through `ux-ui-designer` first — this is at-table interaction design, the same bar Phase 8/11's slot-grid and charge-tapping surfaces were held to.
- **Item hand-off between party members in Live Session. (Must-have.)** `LiveSessionInventoryModal.svelte` can burn a charge but has "deliberately no add/edit/remove/reslot," and no transfer either. "Wren hands Pip the rope" happens constantly in play; today the GM has to track it outside the app or edit two Roster forms mid-session, breaking the one-handed-at-the-table requirement. A minimal "move this item to [other party member]" action from the same modal, respecting the receiving mouse's 10-slot cap, closes this. Pairs naturally with the HP tracker above — post-fight loot handoffs are the common case.

**Explicitly not in Phase 13:** a hex-crawl travel/watch tracker (watches per day, travel checks, foraging, per-watch encounter checks) — Mausritter's actual hex-crawl turn structure isn't modeled at all today, but that's a genuinely new domain concept (game-time/turn tracking), not a rider on this phase. It deserves its own scoped phase with a `ux-ui-designer` pass, flagged here so it doesn't get lost. Also deferred: a global quick-find search across campaign content, and a Dashboard data-safety export reminder — both real, both smaller, both better scoped once Phase 13's must-haves land.

## Cross-cutting, not a phase

- **Multi-campaign / co-GM real-time sync.** No backend by design (`CLAUDE.md`); the "hand off to a co-GM" story is the export/import flow already shipped, not sync.

## A note on unscoped work

Two commits landed after Phase 11 was marked complete (multiple concurrent adventures, campaign renaming) without ever touching this file — the roadmap simply went stale for a beat. They're now documented above, in "Where we are," rather than backfilled as a fake "Phase 11.5." Reasoning: a numbered phase in this document represents a *scoped, prioritized* body of work reviewed against real GM needs before it's built, in the style of every phase above. These two features weren't that — they were built ad hoc, and dressing them up with phase numbering after the fact would misrepresent how they came to exist and set a bad precedent (why scope anything up front if it's the same either way in hindsight). Folding them into the "Where we are" preamble, the way this document already treats history, says what's true: they shipped, they're real, and Phase 12 exists specifically because one of them (concurrent adventures) wasn't finished being threaded through the rest of the app. Going forward, any feature big enough to touch a data model shared across screens (an entity, not a bugfix) should get scoped here *before* it's built, not written up after — that's the actual lesson, not the numbering.
